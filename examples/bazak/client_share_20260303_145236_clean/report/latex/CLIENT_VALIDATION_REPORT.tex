\subsection{Per-Trial Results (Consolidated)}

\textbf{Measure definitions used in this section}

\textit{Raw quality measures}
\begin{itemize}
    \item \texttt{ExA} (Example Accuracy): Per-example accuracy for a single model+example pair.
    Formula per example \(i\): \(\mathrm{ExA}_i =\) mean of numeric per-example metrics whose names end with \texttt{\_accuracy}.
    Important: this mean uses only keys returned for that example (sparse metrics), not a fixed 3-term denominator.
    \item \texttt{Tool Acc}: Trial-level correct tool selection rate.
    \item \texttt{Param Acc}: Trial-level correct tool-parameter rate.
    \item \texttt{Text Acc}: Trial-level text-response correctness rate.
\end{itemize}

\textit{Derived aggregate measures}
\begin{itemize}
    \item \texttt{Accuracy}: Trial-level mean of per-example accuracy:
    \[
    \mathrm{Accuracy}=\frac{1}{N}\sum_{i=1}^{N}\mathrm{accuracy}_i,\quad N=3
    \]
    Here, \(\mathrm{accuracy}_i\) is the example-level \texttt{accuracy} returned for each evaluation example.
    Worked example (\texttt{gemini-2.5-flash}):
    \[
    (1.0 + 1.0 + 0.5) / 3 = 0.8333.
    \]
    In section 3.3, this value is shown as \texttt{Acc} and repeated for each row of the same trial.
    In this run it is \textbf{not} computed as
    \[
    \frac{\texttt{tool\_accuracy}+\texttt{param\_accuracy}+\texttt{text\_accuracy}}{3}.
    \]
    That split-metric mean is used only as a fallback when explicit \texttt{accuracy} is missing.
    \item \texttt{Score}: Same as \texttt{Accuracy} in this run. In section 3.3, shown as \texttt{Scr} and repeated per row of the same trial.
\end{itemize}

\textit{Operational measures}
\begin{itemize}
    \item \texttt{Lat(ms)}: Mean latency over examples in the trial (lower is better).
    \item \texttt{SDK\$}: Total cost across all 3 examples (lower is better).
    \item \texttt{FE\$}: Per-example cost displayed in FE (\texttt{SDK Total Cost / 3} for this run; lower is better).
\end{itemize}

\section{Clarifications for Client Questions}

\subsection{Why FE ``Cost'' Differs from SDK ``Total Cost''}

\begin{itemize}
    \item SDK trial metrics store total trial cost across all 3 examples.
    \item FE ``Cost'' is displayed as a per-example value in this run.
    \item Evidence (all trials): \texttt{FE Cost \textasciitilde{} SDK Total Cost / 3}.
\end{itemize}

This is a representation difference, not a data-integrity issue.

\subsection{Why ``Best Config'' is gemini-3-flash-preview while gemini-2.5-flash has Higher Overall Accuracy}

\begin{itemize}
    \item Optimization objective for this run was \textbf{tool\_accuracy} (single objective).
    \item \textbf{Important note:} \texttt{tool\_accuracy} measures only whether the correct tool was selected. It does not directly reward parameter correctness (\texttt{param\_accuracy}) or final text correctness (\texttt{text\_accuracy}).
    \item gemini-3-flash-preview and gemini-2.5-flash tied on \texttt{tool\_accuracy} (0.6667).
    \item Because this objective produced a tie, selection fell back to deterministic tie behavior for this run and picked gemini-3-flash-preview.
    \item If optimizing for overall \texttt{accuracy}, gemini-2.5-flash would rank highest (0.8333).
\end{itemize}

\subsection{Content Novelty and Content Uniqueness}

Displayed values in FE:
\begin{itemize}
    \item Content Novelty: \texttt{0.5000}
    \item Content Uniqueness: \texttt{0.8336}
\end{itemize}

Meaning:
\begin{itemize}
    \item \textbf{Content Uniqueness} estimates how different each evaluation input is from other inputs in the same dataset. Higher values mean less overlap and repetition.
    \item \textbf{Content Novelty} estimates how far an example is from the dataset center (how atypical it is relative to the rest).
\end{itemize}

Why these are the same across all four trials:
\begin{itemize}
    \item They are dataset-level descriptors, not model-quality metrics.
    \item They are derived from the evaluation examples themselves and then shown consistently across model trials for that run.
\end{itemize}

Interpretation for this run:
\begin{itemize}
    \item \texttt{0.8336} uniqueness indicates the three examples are fairly distinct from each other.
    \item \texttt{0.5000} novelty is a neutral value in this run and should not be used to compare model quality.
\end{itemize}

\subsection{Service-Provider Metric Behavior (Client-Side Handling)}

Live API spot-checks run on \textbf{March 3, 2026} against \texttt{POST /traigent/v1/evaluate} showed the following response behavior:
\begin{itemize}
    \item Per-example metric keys are \textbf{sparse} (different examples may return different subsets of \texttt{tool\_accuracy}, \texttt{param\_accuracy}, \texttt{text\_accuracy}).
    \item The evaluate response does \textbf{not} include explicit per-example \texttt{accuracy}.
    \item Aggregate metric keys can still include \texttt{text\_accuracy} even when that key appears only on a subset of examples.
\end{itemize}

Client-side handling implemented in this run:
\begin{itemize}
    \item Per-example \texttt{ExA} is derived as the mean of available numeric \texttt{*\_accuracy} keys for that example.
    \item Trial-level \texttt{Acc} is the mean of per-example \texttt{ExA}.
    \item Trial-level \texttt{TA} / \texttt{PA} / \texttt{XA} are each averaged over examples where that specific key exists (missing keys are omitted, not treated as zero).
\end{itemize}

Important logging limitation for auditability:
\begin{itemize}
    \item Trial artifacts currently persist per-example \texttt{accuracy}, but not the full per-example split metric map (\texttt{tool\_accuracy} / \texttt{param\_accuracy} / \texttt{text\_accuracy} by example).
    \item This means exact per-example split-key reconstruction is not fully possible from stored trial files alone after the run.
\end{itemize}

Recommended follow-up:
\begin{itemize}
    \item Persist full per-example metric dictionaries in trial logs to make post-run derivations and denominator choices fully auditable.
\end{itemize}
