## Part 1 — Action Summary (order of implementation)

Trigger: a customer/company will not accept `litellm` as a dependency after its March 2026 PyPI supply-chain compromise (backdoored `1.82.7`/`1.82.8`). Traigent's current pin (`>=1.87.1,<2`) is not in the compromised or CVE-affected range, and none of litellm's disclosed 2026 CVEs apply to Traigent (they're all in the standalone LiteLLM Proxy server, which Traigent never runs — Traigent only imports the library). The real, non-dismissible risk is structural: a single high-value package sitting in the credential path for every provider. That's worth actually removing, not just re-pinning.

Traigent has more of the groundwork than it gets credit for, but less than my first pass claimed — corrected below after reading `tests/unit/examples/test_provider_examples.py` and `traigent/examples/providers/manifest.json`, which are the actual source of truth (they're what CI enforces), not just the plugin files in isolation. `litellm` is load-bearing in three places, not two: the pricing source in `cost_calculator.py`; the optional `litellm_plugin.py`/`litellm_interceptor.py` path; and — the part I missed initially — **5 of the 9 officially bundled, tested, documented quickstart examples** (Azure, Gemini, Groq, AWS Bedrock, GCP Vertex) are implemented via `litellm.completion()` today, with a codified test (`test_provider_entry_has_required_keys`) that literally asserts *"LiteLLM is a core dependency, so it must never carry an extra pip package"* for any manifest entry with `impl: "litellm"`. That test itself needs to change as part of this work, not just the code it's testing.

Order of implementation:

1. **Hash-pin the current litellm version and add a CI supply-chain gate** (`pip-audit` / `osv-scanner` on every build). Closes the acute gap immediately regardless of everything else below.
2. **Replace `cost_calculator.py`'s primary pricing source.** Stop relying on litellm's single crowdsourced `model_cost` JSON as ground truth. Use a cross-checked multi-source pricing dataset instead — `genai-prices` (Pydantic, cross-references LiteLLM/OpenRouter/Helicone/llm-prices and flags discrepancies) as primary, with Portkey's open pricing registry (~3,500 models/50+ providers) as a second cross-check source. Keep Traigent's existing fail-fast + custom-pricing-override design as-is — it's the right shape, it just needs a better-fed fallback table instead of the current ~12-model hand-maintained dict.
3. **Confirm `litellm_interceptor.py` no-ops cleanly when litellm isn't installed** — mirror the graceful-absence pattern `cost_calculator.py` already has (`try: import litellm ... except ImportError: LITELLM_AVAILABLE = False`).
4. **Rewrite the Azure, Gemini, and Groq examples to native SDK calls — zero new plugin code needed, verified against the actual plugin target classes:**
   - Azure → `openai.AzureOpenAI(...)`, already intercepted by `azure_openai_plugin.py` (targets `openai.AzureOpenAI`/`AsyncAzureOpenAI`).
   - Gemini (AI Studio) → `google.generativeai.GenerativeModel(...)`, already intercepted by `gemini_plugin.py`.
   - Groq → `openai.OpenAI(base_url="https://api.groq.com/openai/v1", ...)`. No dedicated Groq plugin exists or is needed — `openai_plugin.py` patches the `openai.OpenAI`/`AsyncOpenAI` classes themselves, so any OpenAI-compatible provider reached through those classes is already intercepted regardless of `base_url`. This is also the pattern to reuse for Mistral/Together/Fireworks/local vLLM later — one generic adapter shape, no new plugin work.
5. **Rewrite AWS Bedrock to Traigent's own `BedrockChatClient`** (`traigent/integrations/bedrock_client.py`, wrapping `boto3` directly), already intercepted by `bedrock_plugin.py`. Caveat: that wrapper's docstring says it was built "for paper experiments" and covers Anthropic-on-Bedrock specifically — treat it as needing a quick fitness review for general quickstart use, not a drop-in assumed-ready swap.
6. **GCP Vertex AI is the one genuine gap — decide explicitly, don't let it default.** No plugin today targets the Vertex-specific SDK path (`gemini_plugin.py` only covers the AI-Studio `google.generativeai` classes). Either scope a small `vertex_plugin.py` (same shape as the others) before flipping this example, or explicitly keep the Vertex example on litellm as a named, deliberate exception rather than a leftover.
7. **Update the manifest and its tests to match** — `manifest.json`'s `impl` field for azure/gemini/groq/aws changes away from `"litellm"`; `test_provider_entry_has_required_keys`'s hardcoded `impl in {"langchain", "litellm"}` and its "litellm ⇒ pip_package is None" assertion need a third case; `test_example_file_exists_and_matches_manifest`'s `"litellm.completion(" in source` check needs an equivalent assertion for the native-impl examples. These tests are the actual guardrail here — the change isn't done until they encode the new reality.
8. **Move `litellm` from core `dependencies` to `optional-dependencies.litellm`** in `pyproject.toml`, once steps 4-7 have removed it from every example except the deliberate Vertex exception and the still-useful "LiteLLM (any provider)" catch-all — both of which move behind `pip install traigent[litellm]` as a named opt-in fallback for the long tail, not a silent default. Non-breaking for existing litellm_plugin users.
9. **Do not add an active-gateway dependency (Bifrost/Portkey/Helicone) as a litellm replacement inside the SDK.** That would swap one third-party trust dependency for another that sees 100% of live traffic/prompts/API keys in transit — a larger blast radius than a compromised local pip import, and it directly undermines the reason this change is happening. If a customer wants real-time spend circuit-breaking on their own agent traffic, that's their infra choice to front their own calls with such a gateway — document it as an optional pattern, don't depend on it.
10. **Run this as a `/spine:feature` ChangeSession.** It's cross-cutting (core dependency, public plugin surface, customer-facing examples, codified tests) and there should be an evidence trail for the security stakeholder who raised the original concern.

## Part 2 — SDK Research (Claude, grounded in this repo)

**Dependency status:** `litellm>=1.87.1,<2` is pinned in `pyproject.toml`'s core `dependencies` (not an extra). Comment there notes only the stable public surface is used: `completion`, `acompletion`, `cost_per_token`, `token_counter`, `model_cost`, `ModelResponse`.

**Correction after reading `tests/unit/examples/test_provider_examples.py` + `manifest.json`:** my first pass over-read the plugin files in isolation and concluded native adapters already meant litellm was mostly decorative. The manifest is the actual source of truth (it drives the bundled examples, the anti-drift tests, and — per its own comment — a vendored copy feeds the portal's Quick Start page too), and it shows litellm is the *live, tested, documented* implementation for Azure/Gemini/Groq/AWS/GCP today, with a test that names litellm as a required core dependency by design. The plugins that make each of those provider SDKs mock-interceptable do already exist (confirmed by reading their `get_target_classes()` directly — see Part 1, step 4-5) except for GCP Vertex, so the corrected conclusion is: **most of the rewrite is example-and-test work against infrastructure that already exists, plus one real gap (Vertex)** — smaller than "rewrite 100+ providers," bigger than "just flip an extras flag."

**What's actually mandatory vs. optional:**
| Component | Role | LiteLLM dependence | Removal impact |
|---|---|---|---|
| `cost_calculator.py` | Pricing lookup | Primary source via `litellm.model_cost`/`cost_per_token()`; already has a graceful `ImportError` fallback to `ESTIMATION_MODEL_PRICING` + custom pricing overrides | Needs a non-litellm primary source (see Part 1, step 2) |
| `litellm_interceptor.py` | Monkey-patches `litellm.completion`/`acompletion` to capture token/cost metadata | Only meaningful if user code actually calls litellm | Should already no-op if litellm absent — verify |
| `litellm_plugin.py` | Multi-provider routing via `litellm.completion()` | Is litellm | Already opt-in behavior; just needs to move to an extras group formally |
| `examples/providers/{azure,gemini,groq}.py` + `manifest.json` | Bundled, tested, documented quickstarts | **Live, not decorative** — `impl: "litellm"`, and a test asserts litellm must stay a core (non-extra) dependency | Rewrite to native SDK calls; each target class is already intercepted (verified: `azure_openai_plugin.py`→`openai.AzureOpenAI`, `gemini_plugin.py`→`google.generativeai.GenerativeModel`, `openai_plugin.py`→`openai.OpenAI`/`AsyncOpenAI` covers Groq via `base_url`) |
| `examples/providers/aws.py` | Bundled Bedrock quickstart | Same as above | Rewrite to `BedrockChatClient`; that wrapper is intercepted by `bedrock_plugin.py` but was built for internal paper experiments — verify fitness before treating as ready |
| `examples/providers/gcp.py` | Bundled Vertex AI quickstart | Same as above | **Genuine gap** — no plugin targets the Vertex SDK path today; needs new work or a deliberate kept-exception |
| `anthropic_plugin.py`, `cohere_plugin.py` | Native per-provider integration, instrumenting the official SDKs directly | **None** | Not affected |
| `@optimize` / orchestrator / trial execution | Core execution loop | **None** — delegates to whatever the user's own agent code calls | Not affected |

**Security fact-check:**
- Litellm's 2026 CVEs (`CVE-2026-42208` SQLi, `CVE-2026-35029`/`-35030` auth bypass, SSTI, MCP RCE `CVE-2026-42271`, plaintext-password exposure) are all confirmed by litellm's own security-hardening post to affect **only the standalone LiteLLM Proxy server**, which Traigent never runs. SDK-only usage (`litellm.completion()`) is explicitly unaffected.
- The **March 24, 2026 PyPI supply-chain compromise** (attackers stole publish credentials via a compromised Trivy scanner in litellm's own CI, shipped backdoored `1.82.7`/`1.82.8` wheels for ~2 hours, harvesting OpenAI/Anthropic/Azure keys, cloud credentials, and installing a systemd C2 backdoor) is the one real, structural concern — it applies regardless of Proxy vs. SDK usage, since the payload runs at import/install time. Traigent's current pin is outside the compromised range, but an open `<2` range doesn't protect against a *future* compromise the same way — hence hash-pinning + CI scanning as an immediate step, and moving litellm to optional as the structural fix.

**Bottom line:** not a 100+-provider rewrite, but not a pure reclassification either. The interception infrastructure for Azure/Gemini/Groq/AWS already exists and is verified by class name — that part is example-and-test rewriting, not new engineering. GCP Vertex is the one provider that needs real new work (or an explicit, documented exception). Decoupling `cost_calculator.py`'s pricing source stays the cleanest, lowest-risk, fully-isolated first move regardless of how the provider-example work is sequenced.

## Part 3 — Gemini Research (web deep research, reconciled against the above)

Source: Gemini deep-research doc, "Evaluation of Alternative LLM Cost Tracking and Governance Solutions for Multi-Vendor Python Agent Architectures."

**What it covers:** it frames the space as *Active Gateways* (Bifrost, Portkey, Helicone — intercept live traffic, enforce budgets/rate-limits in real time, e.g. Bifrost's virtual-key monthly caps, Helicone's `Helicone-RateLimit-Policy` header, Portkey's workspace budget locks) vs. *Passive Observability SDKs* (Langfuse, OpenLLMetry — trace-level cost attribution after the fact, via `@observe`/`@weave.op` decorators or OTel auto-instrumentation) vs. *Enterprise APM* (Datadog LLM Observability, W&B Weave, Braintrust — cost folded into a broader monitoring platform, sometimes reconciled against actual provider invoices).

**Reconciliation — this answers a different question than "what should Traigent's SDK depend on":** every tool in that report assumes an architecture where *your own organization's LLM traffic* flows through infrastructure you operate (a gateway in front of your calls, or a tracing SDK wrapping your own agent code). Traigent is a `pip install`ed library that customers' own agent code imports — it does not operate a gateway that customer traffic passes through. Adopting Bifrost/Portkey/Helicone as a *mandatory* replacement inside Traigent would mean forcing every Traigent user's LLM traffic through a new third-party proxy — a bigger architectural imposition than litellm ever was, and it reintroduces the exact problem this whole exercise is trying to remove (a single third-party sees all traffic/keys — arguably worse, since it's live network interception rather than a local import). None of the report's gateway/observability platforms should become an internal Traigent dependency.

**What's actually transferable from it:**
- **Portkey's open-source pricing registry** (~3,500 models/50+ providers) is a legitimate second cross-check source alongside `genai-prices` for hardening `cost_calculator.py`'s pricing data (Part 1, step 2) — same category of fix already planned, just one more independent source to reconcile against.
- **The "active gateway via `base_url` override" pattern** (Bifrost/Helicone/Portkey all work by pointing `base_url` at the gateway) confirms the generic OpenAI-compatible adapter approach (Part 1, step 5) is the right shape — it's the same mechanism, just pointed at the actual provider instead of a proxy.
- **OTel-based export (OpenLLMetry's `gen_ai.usage.*` span conventions)** is worth keeping in mind as a future *optional* integration for customers who already run Datadog/other APM and want Traigent's trial/run cost data to show up there — a roadmap item for `traigent/observability/`, not a litellm replacement.
- **Active spend-limiting (Bifrost/Helicone/Portkey budgets)** is a legitimate customer-side pattern worth documenting ("if you want real-time circuit-breaking on your own agent traffic, front your calls with one of these") but should stay a documented option, not a Traigent dependency — Traigent's own cost tracking is inherently post-hoc/pre-estimation, not a live wire-level enforcement point, and that's fine for what it needs to do.
