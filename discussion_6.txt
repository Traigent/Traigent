## Work Completed

### 1. Event Trace Implementation

**Goal:** Add JSONL event tracing to the paired MC so post-hoc questions ("Which configs
survived P6 in trial 42?") can be answered from logs without re-running.

**Implementation (per PLAN_event_tracing.md):**

Verification before implementation: launched an Explore agent that audited the plan against
`run_demo.py` and confirmed zero behavioral changes — traces are append-only observation.

**Changes to `run_demo.py`:**

- **Elimination helpers** — extended `_masked_empirical_eliminate()` and `_masked_fisher_eliminate()`
  to return `elim_meta` dict with `best_config`, `threshold`, `threshold_mult`. Avoids brittle
  threshold reconstruction in the caller.

- **`_walk_and_probe()`** — added `trace: list[dict] | None`, `phase: str`, `round_num: int`
  parameters. Emits `item_probed` events in the main probing loop (with b_emp, b_star, tau,
  decision, budget_charged) and `item_scored` events in the edge case path (|A| ≤ n_probe).

- **`run_single_trial()`** — initializes `trace = []`, emits all 7 event types:
  - `phase_start` before Phase A, B, each C round
  - `item_scored` in seed loop and non-probing loops
  - `item_probed` via `_walk_and_probe()`
  - `model_fit` after each IRT fit
  - `elimination` after each elimination call (using elim_meta)
  - `early_stop` on single_survivor, epsilon_convergence, permutation_exhausted
  - `winner` at end with surviving_configs, full/unseen accuracies
  Returns trace + surviving_configs in result dict.

- **`run_paired_mc()`** — added `trace_path` parameter. Opens file, writes one JSON line per
  trial-pipeline, flushes after each write. Pops trace from result before storing in memory.

- **`main_paired_mc()`** — added `--no-traces` CLI flag (default: traces ON).

**Verification:** Smoke test (n=5) produced 30 JSONL lines. Spot-checked: winner events have
correct surviving_configs, elimination.eliminated matches set difference, item_probed decisions
match |b_emp - b_star| vs tau.

### 2. Survivor Identity Analysis

**Goal:** Show which configs form a stable "core set" across MC trials vs random "fringe".

**New functions in `run_demo.py`:**
- `survivor_identity_analysis(results)` — computes per-pipeline: |A| stats (median/min/max),
  core set (configs surviving ALL trials), fringe set (some but not all), mean pairwise Jaccard
  similarity, per-config survival frequency.
- `print_survivor_identity_table(identity, n_trials)` — terminal table.

**New in `visualize.py`:**
- `plot_survivor_heatmap(data, out_dir)` — config × pipeline heatmap (YlOrRd, annotated with %).
- Wired into `generate_paired_mc_figures()`.

**Finding (n=5):** Core set of 6 configs {4,5,9,12,13,17} shared across P2/P5/P6. P3 barely
eliminates anything (median |A|=18). P6 most aggressive (median |A|=8).

### 3. Survivor Quality on Test Set

**Goal:** Score survivors on the held-out test set (100 items), not train "unseen" items.

**Context:** User corrected my initial approach — "unseen" train items (items the pipeline didn't
touch) are NOT the same as the held-out test set. The test set (`score_matrix_test.npy`, 20×100)
is the proper evaluation.

**New functions in `run_demo.py`:**
- `survivor_quality_analysis(results, test_matrix, bf_winner)` — per-pipeline: winner test acc,
  best survivor test acc, BF kept fraction, avg survivor test acc, # survivors ± std.
- `print_survivor_quality_table(quality)` — terminal table.

**Finding (n=5):** Config 5 (gpt-4o/cot) is test-set best at 80%, while config 12 (gpt-4o/fewshot)
is train-set best at 72% train / 78% test. All pipelines keep BF winner in all 5 trials.

### 4. Elimination Curve Analysis

**Goal:** Show how many evals each pipeline needs to reach key |A| milestones (15, 10, 5, 3, 1).

**New functions in `run_demo.py`:**
- `elimination_curve_analysis(trace_path)` — reads JSONL, walks events, sums evals_charged from
  item_scored/item_probed events, records (cumulative_evals, len(active_after)) at each
  elimination. Aggregates across trials per pipeline per threshold.
- `print_elimination_curve_table(curve)` — terminal table with median evals and % of trials
  that reached each threshold.

**Wired into `main_paired_mc()`** — runs automatically when traces are available.

**Finding (partial n=384 from in-progress run):**
- P3 never gets below |A|=15 — barely eliminates
- P6 reaches |A|≤10 in 89% of trials vs P2's 16% — targeting advantage
- No pipeline reliably reaches |A|=1 — they stop by epsilon convergence

### 5. Timestamped Run Directories

**Problem:** Smoke tests overwrote the n=500 results file. Each `--smoke` run wrote to the
same `results/paired_mc_results.json`, destroying the 75-minute n=500 run's output.

**Fix:** Each run now writes to `results/paired_mc/n{trials}_{YYYYMMDD_HHMMSS}/`:
```python
from datetime import datetime
run_tag = datetime.now().strftime("%Y%m%d_%H%M%S")
run_dir = RESULTS_DIR / "paired_mc" / f"n{n_trials}_{run_tag}"
```

Updated `visualize.py` with `_find_latest_paired_mc_dir()` to find the most recent run.

## Key Findings

1. **Tracing adds ~2x overhead** — n=500 traced run takes ~70 min vs ~35 min untraced
   (not 4x as initially feared; the 4x estimate was from comparing against an overly
   optimistic baseline).

2. **P6 has a stable core of 6 configs** — {4,5,9,12,13,17} survive in all trials.
   Remaining survivors are "fringe" — appearing in some trials but not all.

3. **P3 barely eliminates** — median |A|=18-19 out of 20 configs. Empirical CIs are
   too wide with 90 items to eliminate confidently.

4. **All 5 comparisons significant at n=500:**
   - P2 vs P3: −215 evals (Fisher saves 12% over empirical) ***
   - P1 vs P2: +709 evals (free accounting inflates cost) ***
   - P4 vs P2: −833 evals (targeting + honest accounting saves 54%) ***
   - P5 vs P2: −30 evals (proportional saves 2%) *** (was n.s. at n=10)
   - P6 vs P5: −71 evals (partial data saves 5%) *** (was n.s. at n=10)

5. **Elimination curve reveals targeting advantage** — P6 reaches |A|≤10 in 89% of trials
   at ~979 evals, while P2 reaches it in only 16% at ~1005 evals.
