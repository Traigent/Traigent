# Product Requirements Document - talmudic-sources-reference-extraction-pipeline

**Author:** Oren
**Date:** 2025-01-20

## Executive Summary

### Product Vision

Talmudic Sources Reference Extraction Pipeline הוא כלי אוטומטי המבוסס על סוכני LLM, שמטרתו לזהות ולחלץ הפניות בין קטעי טקסט קנוניים בספרות התלמודית. הכלי סורק טקסטים, מזהה הפניות מפורשות וסמויות בין מקורות, ומייצא את הממצאים לקובץ JSON להעלאה לגרף הידע.

### The Problem

גרף הידע של הספרות התלמודית חסר קשרים קריטיים בין קטעי טקסט. הספרות התלמודית לדורותיה מכילה רשת עצומה של הפניות - חלקן מפורשות וחלקן סמויות - שאינן ממופות כיום. ללא מיפוי הקשרים הללו, לומדי תורה ומערכות AI מחמיצים הקשרים חיוניים להבנת הטקסטים.

### Target Users

- **Primary:** מפתחים בצוות TrueLogic שמעלים את פלט ה-JSON לגרף הידע
- **End Beneficiaries:** לומדי תורה שייהנו מגרף ידע מועשר, ומערכות AI שישתמשו בגרף לענות על שאלות משתמשים

### What Makes This Special

1. **קנה מידה חסר תקדים** - עיבוד ~250 מיליון מילים מטקסטים תלמודיים שונים
2. **זיהוי הפניות סמויות** - LLM Agents מזהים גם הפניות שאינן כתובות מפורשות
3. **פתרון ראשון מסוגו** - אין כלי קיים שפותר את הבעיה הזו
4. **ערך כפול** - העשרת גרף הידע משרתת גם לומדי תורה וגם את ה-AI של TrueLogic

## User Journeys

### Journey 1: דני - כיול Benchmark וניקוד אוטומטי

דני הוא מפתח Pipeline בצוות TrueLogic שמופקד על פיתוח כלי חילוץ ההפניות. לפני שהוא מריץ את הכלי על קורפוס שלם, הוא צריך לוודא שהקונפיגורציה שלו (פרומפטים, מודל, פרמטרים) עובדת כמו שצריך.

דני בוחר טקסט קטן ומוגדר - למשל פרק אחד מהריטב"א על מסכת קידושין - שיש לו ground truth ידוע (רשימת הפניות מאומתת ממהדורה מדעית). הוא מריץ את ה-Pipeline במצב benchmark, והמערכת מחלצת את ההפניות מהטקסט ומשווה אותן אוטומטית ל-ground truth.

המערכת מציגה לדני ניקוד מפורט:

- **Precision**: 87% - רוב ההפניות שזוהו אכן נכונות
- **Recall**: 72% - נמצאו רוב ההפניות, אבל כמה הוחמצו
- **Cost**: 1,250 טוקנים לעמוד, $0.03 לעמוד

דני רואה שה-Recall נמוך מדי. הוא משנה את הפרומפט להיות יותר רגיש להפניות סמויות, מריץ שוב את ה-benchmark, ומקבל:

- **Precision**: 82%
- **Recall**: 89%
- **Cost**: 1,800 טוקנים לעמוד, $0.045 לעמוד

עכשיו יש trade-off ברור: יותר Recall אבל יותר עלות. דני מחליט שזה מקובל ומתקדם להרצה מלאה.

### Journey 2: דני - הרצה בסקייל מלא על הקורפוס

אחרי שדני כייל את ה-Pipeline עם ה-benchmark והשיג ציונים מספקים, הוא מוכן להריץ על הקורפוס המלא. הוא בוחר את פרשנויות הריטב"א על כל התלמוד - אלפי עמודים.

דני מפעיל את ה-Pipeline במצב production. המערכת מתחילה לעבד טקסט אחר טקסט, מציגה התקדמות, ושומרת את ההפניות המזוהות לקובץ JSON. אחרי כמה שעות, העיבוד מסתיים עם דוח סיכום:

- 12,450 הפניות זוהו
- עלות כוללת: $156
- שגיאות: 3 טקסטים נכשלו (יטופלו בנפרד)

דני בודק מדגם מהתוצאות ורואה שהאיכות עקבית עם מה שראה ב-benchmark. הוא מעביר את קובץ ה-JSON לצוות הגרף.

### Journey 3: יוסי - בדיקת דגימה מול מהדורה מדעית

יוסי הוא מומחה לתלמוד בצוות TrueLogic. אחרי שדני מריץ את ה-Pipeline על קורפוס, יוסי מבצע בדיקת איכות מדגמית. הוא בוחר 10 קטעים אקראיים מהפלט ומשווה אותם למהדורה מדעית שיש לו בספרייה.

יוסי פותח את קובץ ה-JSON, בוחר קטע מהריטב"א על מסכת גיטין, ומשווה להפניות שמופיעות במהדורת מוסד הרב קוק. הוא מגלה שהכלי מצא 8 מתוך 9 הפניות, אבל גם זיהה הפניה אחת שנראית לו שגויה.

יוסי מתעד את הממצאים: "Recall טוב, אבל יש נטייה לזהות ביטויים נפוצים כהפניות". הוא מעביר את המשוב לדני.

### Journey 4: דני - תיקון פרומפטים אחרי משוב

דני מקבל את המשוב מיוסי. הוא קורא את הדוגמאות הספציפיות ומזהה pattern - הכלי מזהה ביטויים כמו "כדאיתא" (כמו שמופיע) כהפניה גם כשזה ביטוי כללי.

דני פותח את קובץ הקונפיגורציה ומשנה את הפרומפט: מוסיף הנחיה לא לזהות ביטויי קישור נפוצים כהפניות בפני עצמם, אלא רק כשהם מלווים במיקום ספציפי.

הוא מריץ שוב את ה-benchmark על אותו טקסט מכויל, ורואה שה-Precision עלה ל-91% בלי פגיעה משמעותית ב-Recall. הוא מסמן את הקונפיגורציה החדשה כ-"v2" ומתכנן הרצה חוזרת על הקורפוס.

### Journey 5: מיכל - העלאה לגרף הידע

מיכל היא מפתחת גרף הידע בצוות. היא מקבלת מדני קובץ JSON עם 12,450 הפניות מזוהות. תפקידה להעלות את ההפניות לגרף ולוודא שהן משתלבות נכון.

מיכל פותחת את הקובץ ובודקת את המבנה: כל הפניה מכילה מקור, יעד, סוג (מפורשת/סמויה), ורמת ביטחון. היא מריצה script שמתרגם את ה-references לפורמט של הגרף ומעלה אותם.

אחרי ההעלאה, מיכל מריצה queries לבדיקה: "תראה לי את כל ההפניות מהריטב"א לתלמוד בבלי". היא רואה רשת קשרים חדשה שלא הייתה קיימת קודם. היא מעדכנת את הצוות שההעלאה הושלמה בהצלחה.

## Developer Tool Specific Requirements

### סקירת סוג הפרויקט

הכלי הוא CLI Pipeline בשפת Python, המיועד להרצה מהטרמינל עם קונפיגורציה מבוססת קבצים. הכלי תומך במספר ספקי LLM ומאפשר גמישות בהגדרות.

### מבנה CLI

**פקודות עיקריות:**

| פקודה | תיאור | דוגמה |
|-------|-------|-------|
| `benchmark` | הרצת benchmark על טקסט מכויל | `pipeline benchmark --config v2.yaml` |
| `run` | הרצת חילוץ על קורפוס | `pipeline run --corpus ritva --output results.json` |
| `config` | ניהול קונפיגורציות | `pipeline config show`, `pipeline config validate` |

**Flags נפוצים:**

- `--config` - נתיב לקובץ קונפיגורציה (ברירת מחדל: `config/default.yaml`)
- `--output` - נתיב לקובץ פלט
- `--verbose` - פלט מפורט
- `--dry-run` - הרצה ללא שמירה (לבדיקה)

### סכמת קונפיגורציה

**מבנה קובץ YAML:**

```yaml
# config/default.yaml
llm:
  provider: "anthropic"  # או "openai"
  model: "claude-sonnet-4-20250514"
  temperature: 0.2
  max_tokens: 4096

extraction:
  prompts:
    explicit_references: |
      [פרומפט לזיהוי הפניות מפורשות]
    implicit_references: |
      [פרומפט לזיהוי הפניות סמויות]
  confidence_threshold: 0.7

output:
  format: "json"
  include_metadata: true
  include_source_text: false

benchmark:
  ground_truth_path: "data/ground_truth/"
  metrics: ["precision", "recall", "f1", "cost"]
```

**היררכיית קונפיגורציה:**

1. קובץ ברירת מחדל (`config/default.yaml`)
2. קובץ ספציפי להרצה (`--config custom.yaml`)
3. Flags בשורת הפקודה (עדיפות גבוהה ביותר)
