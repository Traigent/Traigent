TraiGent SDK Development Session - December 17, 2025
=====================================================

SESSION SUMMARY
---------------
This session focused on debugging CSV pass/fail reporting and ensuring
tests import production code rather than maintaining duplicate copies.

KEY ISSUES FIXED
----------------

1. CSV PASS/FAIL BUG
   Problem: All results showed PASS even when they should FAIL

   Root Cause: The CSV code was using `ex_result.success` which just indicates
   whether the LLM API call succeeded (didn't throw an error), NOT whether the
   answer passed the evaluator.

   Fix: Changed to use `metrics.get("accuracy")` which contains the actual
   evaluator score. Pass/fail is now determined by score >= 0.5.

   Files changed:
   - examples/quickstart/01_simple_qa_real_llm.py
     * save_results_to_csv() - now reads metrics["accuracy"] for pass/fail
     * print_category_summary() - same fix applied

2. TEST FILE HAD DUPLICATE SCORER CODE
   Problem: test_custom_accuracy_scorer.py had its OWN COPY of the scorer
   function instead of importing from production code. This means tests
   validated the test file's copy, not the actual scorer used by TraiGent.

   Fix: Created shared module examples/quickstart/scorers.py that both
   the example and test import from.

   Files created/changed:
   - examples/quickstart/scorers.py (NEW - single source of truth)
   - examples/quickstart/01_simple_qa_real_llm.py (imports from scorers)
   - examples/quickstart/test_custom_accuracy_scorer.py (imports from scorers)

   New rule added to AutoCoder_orders.txt:
   "7. TESTS MUST IMPORT PRODUCTION CODE"

DEEPER ISSUE DISCOVERED (NOT FIXED)
-----------------------------------
The TraiGent core appears to have a bug in how it stores evaluator scores.

Evidence from CSV results:
- "William Shakespeare" in "William Shakespeare." -> score=0.0 (should be 1.0)
- "1945" in "World War II ended in 1945." -> score=0.0 (should be 1.0)
- "Paris" in "Paris." -> score=0.0 (should be 1.0)

Our custom_accuracy_scorer does: `expected.lower() in output.lower()`
This should return 1.0 for all the above cases.

Hypothesis: TraiGent is either:
1. Not calling the custom evaluator at all
2. Using a different metric for "accuracy"
3. Has a bug in how it stores/retrieves custom evaluator scores

This needs investigation in TraiGent core code.
