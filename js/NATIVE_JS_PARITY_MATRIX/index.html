    Native JS Parity Matrix¶

    This matrix describes the verified state of the native-first JS checkout in this repository root.

    Use it together with:

    • Python SDK Module Catalog and Gap Analysis
    • JS Parity Roadmap
    • Hybrid JS parity status is summarized in docs/HYBRID_JS_PARITY_MATRIX.md.

    Important scope note:

    • this file is about the native execution path in the current checkout
    • call-level backend-guided hybrid Optuna exists in this repo, but spec-level execution.mode = "hybrid" is still rejected

    Label Semantics¶

    • matched: implemented in this checkout and covered by passing tests or a verified public example
    • partial: implemented with bounded semantics and covered, but still behind Python
    • gap: no backend change is required; JS could implement it in this project today
    • deferred-backend: blocked on hybrid/backend protocol or control-plane work
    • out-of-scope: not a target for the native JS SDK

    Capability Python Native JS Overall JS Evidence Notes

    Python remote cloud-execution control plane Yes out-of-scope out-of-scope docs/HYBRID_JS_PARITY_MATRIX.md JS does not target Python's server-side /agent/optimize-style remote execution model; the product boundary is backend-guided local execution.

    Explicit Non-Goals for the Native Checkout¶

    • full cloud/session control-plane parity
    • Python-style remote cloud execution and server-side agent reconstruction
    • Python’s platform/security/analytics families
    • global get_config() semantics
    • full Optuna-family parity in local/native JS
