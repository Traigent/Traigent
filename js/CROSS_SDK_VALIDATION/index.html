    Cross-SDK Validation¶

    Native JS correctness and performance are validated against Python-owned fixtures and benchmark specs in the sibling Traigent repo.

    Correctness Oracles¶

    Python exports deterministic oracle payloads from:

    • ../Traigent/tests/cross_sdk_oracles/generate_native_js_oracles.py

    The JS parity tests consume that payload in:

    • tests/cross-sdk/oracle.test.ts

    Covered oracle cases:

    • grid_3x3: exact config order, best config, best metrics, normalized stop reason
    • random_seed_42: exact seeded config sequence for deterministic discrete sampling
    • budget_cutoff: same budget stop point, total cost, best config, normalized stop reason
    • bayesian_branin: optional outcome-envelope check when the Python Bayesian environment is available

    If the sibling Python repo is absent, the cross-SDK oracle test skips locally. Internal release validation must run it.

    Async Benchmark Methodology¶

    Cross-language async comparison uses a synthetic workload, not real provider calls.

    Shared benchmark inputs come from:

    • ../Traigent/tests/cross_sdk_oracles/generate_native_js_oracles.py
    • ../Traigent/tests/cross_sdk_oracles/run_async_scheduler_benchmark.py
    • scripts/run-cross-sdk-benchmark.mjs

    Method:

    1. Use the same seeded random config space and the same sleep schedule in both runtimes.
    2. Run warmups first, then measured runs at concurrency 1, 2, 4, and 8.
    3. Compare normalized scheduler behavior rather than absolute language speed.

    Reported metrics:

    • wall-clock time
    • throughput
    • theoretical minimum makespan
    • normalized overhead ratio
    • duplicate-config rate
    • context-leak count
    • RSS delta
    • best config
    • best latency
    • stop reason
    • trial count

    Commands¶

    npm run check:schema-types
npm run check:version
npm run test:cross-sdk
npm run benchmark:cross-sdk

    The benchmark command builds the SDK, runs the JS benchmark harness, executes the Python benchmark script, and prints a combined JSON report.

    Hybrid Session Contract Checks¶

    Hybrid JS optimization adds mocked contract validation around the Python backend session API:

    • session creation payload shape
    • next-trial suggestion decoding
    • result submission payload shape
    • stop-reason normalization

    These checks live in:

    • tests/unit/optimization/hybrid.test.ts

    Manifest-Backed Product Client Parity Checks¶

    The authoritative Python/JS parity manifest is owned by TraigentSchema:

    • ../TraigentSchema/parity/python-js-sdk.json

    CI checks out TraigentSchema at an explicit release tag rather than main so schema changes do not silently change JavaScript CI. Refresh that pinned ref when a new schema release is intentionally adopted by the JS SDK.

    For JS 0.2.0 it pins Python origin/develop at f9f1adcb19ea8c3874bb1b7ef21f6d11b1b95a18. It intentionally does not pin a JS HEAD SHA; every JS build is valid if it satisfies the manifest-required exports. JS 0.2.0 may ship before Python 0.12.0, but if it does, release notes must call that out as forward-compatible parity with the next Python surface rather than parity with currently published Python 0.11.4.

    The product client surface is generated from the canonical ../TraigentSchema/traigent_schema/schemas contracts and exposed through:

    • @traigent/sdk/projects
    • @traigent/sdk/prompts
    • @traigent/sdk/evaluation
    • @traigent/sdk/observability
    • @traigent/sdk/core-metrics
    • @traigent/sdk/admin
    • @traigent/sdk/schema

    npm run check:schema-types fails when generated TS contract types drift from TraigentSchema. npm run check:parity-manifest fails when the JS package root does not expose the symbols TraigentSchema currently requires for Python develop parity. npm run check:version keeps runtime telemetry headers aligned with package.json. The package smoke suite also imports every new subpath through both ESM and CJS so package export drift is caught before release.

    The older Python-side ignore-list guard should be replaced by this manifest contract rather than kept as a second source of truth. Python CI can still read the JS API snapshot, but TraigentSchema owns the per-symbol classification: matched, additive-js-alias, deferred-schema, deferred-backlog, or out-of-scope-permanent.

    Current explicit 0.2.0 deferrals:

    • BenchmarkClient is a schema-deferred JS stub until the dataset-generation endpoint is schema-owned (Traigent/traigent-js#31).
    • EnterpriseAdminClient remains a JS migration stub pending Traigent/traigent-js#30. Enterprise admin operations are REST-only in 0.2.x.
    • observability default-client helpers (get_default_observability_client, set_default_observability_client) and the observe wrapper are deferred to 0.3.0 at the earliest (Traigent/traigent-js#38); none of them are exported as stubs in 0.2.x.
    • catalog recommendation APIs, strategy presets, add_agent_span-style instrumentation helpers, advisory selection helpers, and the onboarding CLI are deferred backlog items rather than JS 0.2.x parity claims.

    The contract-policy detail and release-gating criteria for these deferred surfaces live in MIGRATION_FROM_PYTHON.md. A JS release that flips any of them from deferred-schema / deferred-backlog to matched in ../TraigentSchema/parity/python-js-sdk.json must land the schema, the Python __all__ change, the JS implementation, and the manifest reclassification together. The parity-manifest gate (npm run check:parity-manifest) is the load-bearing check that keeps these three surfaces aligned across SDKs.

    Manifest refresh cadence:

    • every Python public SDK release candidate or release branch
    • every Python PR family that changes traigent.__all__
    • before every JS minor parity release
    • quarterly drift check if neither SDK has released in the previous quarter
