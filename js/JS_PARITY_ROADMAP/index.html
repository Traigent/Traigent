    JS Parity Roadmap¶

    This roadmap turns the current Python-vs-JS gap analysis into an execution order.

    Use this together with:

    • NATIVE_JS_PARITY_MATRIX.md
    • PYTHON_SDK_MODULE_CATALOG_AND_GAP_ANALYSIS.md
    • Hybrid JS parity status is summarized in docs/HYBRID_JS_PARITY_MATRIX.md.

    Important branch note:

    • this checkout supports native JS optimization plus call-level backend-guided hybrid Optuna through typed sessions
    • spec-level execution.mode = "hybrid" is still rejected
    • Python's separate server-side remote cloud-execution model is not a JS SDK parity target unless the product direction changes

    Status Label Rules¶

    • matched: implemented and covered by passing tests or a verified public example
    • partial: implemented with bounded semantics and covered, but still behind Python
    • gap: the backend/API contract is reachable today and the JS side could implement it now
    • deferred-backend: blocked on missing or insufficiently specified backend/protocol support
    • out-of-scope: not a current JS SDK target

    The 0.2.0 public root parity gate uses the schema-owned manifest at ../TraigentSchema/parity/python-js-sdk.json. That manifest has a more precise per-symbol vocabulary:

    • matched
    • additive-js-alias
    • deferred-schema
    • deferred-backlog
    • out-of-scope-permanent

    It pins Python origin/develop at f9f1adcb19ea8c3874bb1b7ef21f6d11b1b95a18 and intentionally does not pin a JS HEAD SHA.

    For 0.2.0, JS may ship before Python 0.12.0. If that happens, the release must describe the manifest as forward-compatible with the next Python public surface, not as parity with published Python 0.11.4.

    Prioritization Rules¶

    Gaps are ranked by:

    1. user-visible parity impact
    2. whether the gap belongs in this branch
    3. whether the divergence is justified
    4. implementation leverage across multiple Python module families

    P0: Highest Product Gap, But Not For This Branch¶

    These are the largest remaining Python-vs-JS gaps overall, but they should be implemented through reachable backend/session contracts, not by reviving the old Python-orchestrated JS bridge.

    Backend-guided hybrid execution¶

    • Python status:
    • full cloud/hybrid/session stack
    • includes a separate remote cloud-execution model
    • JS status:
    • call-level backend-guided hybrid Optuna exists through .optimize({ algorithm: "optuna", ... })
    • typed session lifecycle helpers are exported
    • high-level plain-agent local evaluation plus backend-guided config suggestion is implemented
    • seamless framework interception is supported for OpenAI / LangChain / Vercel AI targets
    • spec-level execution.mode = "hybrid" is intentionally rejected here
    • Why it matters:
    • this is the main JS parity path for backend-owned Optuna-like functionality
    • JS parity targets backend-guided local execution over typed /sessions, not Python's server-side remote execution model

    Cloud/session/control-plane parity¶

    • Python status:
    • remote clients
    • session lifecycle
    • subset selection
    • billing/privacy/auth DTOs
    • JS status:
    • low-level typed session helpers for create / next-trial / submit-result are exported
    • list / health / status / finalize / delete helpers normalize DTOs
    • executable end-to-end session-control examples exist
    • still missing parts of Python's wider cloud/control-plane breadth that are either backend-dependent or intentionally out of scope for JS
    • Why it matters:
    • additional typed /sessions parity should be limited to reachable backend routes with known contracts
    • Python's remote cloud-execution family is not a JS target unless the product direction changes

    P1: Highest-Value Native Gaps In This Branch¶

    These are the most important justified parity gaps that still belong in this native-first checkout.

    TVL promotion and statistical semantics¶

    • Current JS status:
    • focused TVL loader
    • artifact-specific nativeCompatibility reporting on loaded TVL artifacts, including usedFeatures and warnings
    • banded objectives
    • safe-expression constraint compilation
    • minEffect, tieBreakers
    • sample-based paired promotion
    • TOST-style band promotion
    • behavioral chanceConstraints
    • Remaining gap:
    • no full promotion-gate lifecycle semantics beyond best-trial selection and post-trial rejection
    • no full TVL CLI/models/runtime parity beyond the supported native subset
    • Priority reason:
    • this is the biggest native-local parity gap still left
    • it affects spec compatibility, promotion semantics, and cross-SDK behavior

    Auto framework override ergonomics¶

    • Current JS status:
    • explicit wrappers exist for OpenAI, LangChain, and Vercel AI
    • seamless mode defaults autoOverrideFrameworks to all active wrapped targets
    • frameworkTargets can narrow active targets and autoOverrideFrameworks: false disables framework interception entirely for a seamless function
    • autoWrapFrameworkTarget(...), autoWrapFrameworkTargets(...), and bounded explicit-object discovery via discoverFrameworkTargets(...) / prepareFrameworkTargets(...) now reduce setup friction for supported framework objects without scanning arbitrary module/global state
    • seamlessResolution() surfaces the resolved seamless path for an optimized function, while describeFrameworkAutoOverride(...) and prepareFrameworkTargets(...).autoOverrideStatus expose active framework target diagnostics
    • Remaining gap:
    • JS still does not auto-discover and wrap framework clients implicitly inside arbitrary user code
    • Guardrails:
    • any future implicit discovery must be opt-in
    • it must not patch globals
    • activation must be visible in logs/diagnostics
    • behavior must be deterministic across import order
    • Priority reason:
    • high user impact
    • closes a real adoption gap without requiring cloud work

    Broader TVL authoring/runtime parity¶

    • Current JS status:
    • native subset is implemented and tested
    • native CLI inspection now exists via traigent inspect tvl ...
    • Remaining gap:
    • full TVL language/runtime parity
    • tighter alignment with Python authoring expectations
    • Priority reason:
    • important if cross-SDK spec portability is a product requirement

    P2: Valuable, But Secondary¶

    These are useful parity improvements, but they are lower priority than P1.

    Example concurrency¶

    • Python status:
    • supports broader evaluation parallelism
    • JS status:
    • opt-in per-example concurrency is available through execution.exampleConcurrency
    • Why it is secondary:
    • the bounded local path now exists, but broader Python evaluation parallelism still exceeds this checkout

    Broader optimizer family parity¶

    • Python status:
    • richer optimizer set, including Optuna-family modes
    • JS status:
    • native grid/random/Bayesian only
    • Why it is secondary:
    • if the long-term strategy is backend-guided hybrid for advanced optimizers, native-local JS does not need to mirror the full Python optimizer family

    Richer trial taxonomy and pruning semantics¶

    • Python status:
    • more detailed failed/pruned/rejected distinctions
    • JS status:
    • simpler native trial outcome model
    • Why it is secondary:
    • useful for introspection, but not the main user-facing parity blocker

    Global config access parity¶

    • Python status:
    • global-style get_config() semantics
    • JS status:
    • intentional wrapper-local / context-local model instead
    • Why it is secondary:
    • parity exists as a mental-model difference, but the JS divergence is partly intentional and arguably better aligned with JS runtime behavior

    P3: Real Python Capabilities, But Not Yet Justified To Copy¶

    These appear in the Python audit, but they should not drive near-term JS SDK work in this repo.

    Platform / enterprise families¶

    • security
    • auth
    • tenancy
    • session hardening
    • analytics/meta-learning

    These are platform-side concerns, not core local-SDK parity targets.

    Assisted generation and discovery families¶

    • bounded tuned-variable discovery is now available in the native checkout through:
    • discoverTunedVariables(...)
    • discoverTunedVariablesFromSource(...)
    • discoverTunedVariablesFromFile(...)
    • traigent detect tuned-variables ...
    • but broader tuned-variable dataflow/range generation and the larger config generation pipelines remain separate product layers from the local native JS optimizer/runtime
    • benchmark/objective recommendation remains out of scope in this checkout
    • the AST parser infrastructure already exists in src/seamless/; the missing tuned-variable work is dataflow/range inference, not new parser plumbing

    Server / wrapper / adapter families¶

    • wrapper service/server
    • hooks
    • execution adapters
    • bridge/server infrastructure

    These should be treated as separate surfaces, not copied blindly into the native-first JS SDK package.

    Recommended Execution Order¶

    Native branch¶

    1. finish TVL promotion/statistical parity
    2. improve auto framework override ergonomics
    3. decide whether full TVL parity is a product requirement or keep a documented supported subset
    4. tighten and benchmark the bounded execution.exampleConcurrency path if runtime throughput becomes a priority

    Hybrid/session surface¶

    1. extend backend-guided hybrid execution parity where reachable backend routes exist
    2. extend the current typed session helper surface only for reachable backend routes with known contracts
    3. align reachable remote DTO/session contracts with Python
    4. only then revisit broader control-plane or advanced optimizer breadth through the backend path

    Non-Goals For This Branch¶

    • implementing cloud/session clients directly in this checkout
    • implementing Python-style remote cloud-executed optimization or server-side agent reconstruction in JS
    • copying platform/security/analytics modules into the JS SDK package
    • forcing Python-style global config semantics into JS if the current wrapper- local/context-local design remains clearer

    0.2.0 Release Parity Decisions¶

    • JS 0.2.0 is forward-bound to Python develop unless Python 0.12.0 ships first.
    • JS 0.2.0 may ship before Python 0.12.0, but release notes must call out that users on Python 0.11.4 will not see the same root symbols yet.
    • BenchmarkClient / BenchmarkClientConfig are root-exported schema-deferred stubs.
    • Dataset and EvaluationExample are root and evaluation-subpath runtime exports.
    • EnterpriseAdminClient remains a throwing migration stub pending Traigent/traigent-js#30.
    • observability default-client helpers and observe move to 0.3.0.
    • the old Python-side public client ignore list should be replaced by the schema-owned manifest instead of duplicated.
    • manifest refresh happens on Python release candidates/release branches, Python traigent.__all__ changes, JS minor parity releases, and quarterly drift checks when release cadence is slow.

    Deferred Contract Decisions And Release Gating¶

    The contract-policy detail for the three deferred surfaces lives in MIGRATION_FROM_PYTHON.md. The summary for roadmap planning is:

    Surface Issue Current JS state Earliest target Gate to flip from deferred-* to matched
    EnterpriseAdminClient Traigent/traigent-js#30 throwing stub; REST-only migration path not in 0.2.x TraigentSchema-owned tenant/membership/SSO contract + Python __all__ export + cross-SDK oracle
    BenchmarkClient Traigent/traigent-js#31 throwing schema-deferred stub not in 0.2.x naming decision recorded + TraigentSchema-owned dataset-generation endpoint + parity-manifest reclass
    get_default_observability_client / set_default_observability_client / observe Traigent/traigent-js#38 not implemented, not exported 0.3.0 at earliest agreed default-client scope (per-process vs context-local) + observe semantics + tests

    No JS minor release should ship any of these surfaces as matched without the parity-manifest reclassification landing in the same PR family. Adding a throwing stub for the observability helpers in 0.2.x is also out of scope, because it would commit the SDK to a runtime shape before the semantics are agreed.
