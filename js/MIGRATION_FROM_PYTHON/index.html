    Migrating From The Python SDK¶

    This JS parity surface targets the canonical Python SDK in the Traigent repo. For the 0.2.0 parity release, the checked-in manifest pins the Python 0.12.0 public surface from the Traigent repo, not the older ../Traigent-reasoning v0.9.0 snapshot. Python SDK 0.12.0 has shipped and removed the Python-orchestrated JS bridge; new JS users should use this repo's docs/getting-started/minimal-integration.md and this migration guide.

    Release Target¶

    The JS parity work is intended for the next additive minor release of @traigent/sdk, 0.2.0. Existing optimization APIs remain backward compatible.

    Client Modules¶

    Python imports map to JS subpath exports:

    Python module JS subpath Main JS client
    traigent.projects @traigent/sdk/projects ProjectManagementClient
    traigent.prompts @traigent/sdk/prompts PromptManagementClient
    traigent.evaluation @traigent/sdk/evaluation EvaluationClient
    traigent.observability @traigent/sdk/observability ObservabilityClient
    traigent.core_metrics @traigent/sdk/core-metrics CoreMetricsClient
    traigent.enterprise @traigent/sdk/admin EnterpriseAdminClient

    These clients are also exported from @traigent/sdk.

    These clients target Traigent platform product APIs. They do not change the Hybrid Mode app-service REST contract used by external services implementing /config-space, /execute, and /evaluate.

    EnterpriseAdminClient is exported as a migration stub because Python exposes it at package root today. Calling it throws a clear "not implemented in JS" error; use the Traigent REST API for enterprise admin operations until that surface is implemented. The implementation decision is tracked in Traigent/traigent-js#30 and the current policy is in Deferred Client Policy.

    BenchmarkClient and BenchmarkClientConfig are also exported from the package root because Python develop re-added them. They are schema-deferred stubs in JS 0.2.0: calling generate(...) or generateSync(...) throws until TraigentSchema owns the benchmark dataset-generation endpoint contract. The product naming decision is tracked in Traigent/traigent-js#31; see Deferred Client Policy for gating criteria.

    Dataset and EvaluationExample are lightweight JS runtime classes that accept both camelCase JS fields and Python-style input_data / expected_output fields. They are available from @traigent/sdk and @traigent/sdk/evaluation.

    Configuration¶

    configure(options) updates process-wide JS SDK defaults. It is additive and idempotent: a second call merges over the previous defaults.

    initialize(options) merges options into the global defaults and resolves a snapshot once. Clients constructed afterward use that snapshot unless a per-client option overrides it. If configure(...) is called after initialize(...), the new options are merged into the bound snapshot; later environment changes still do not leak in.

    resetTraigentConfig() clears configured defaults and any initialized snapshot. It is primarily intended for tests and isolated scripts.

    configure({ field: undefined }) is a no-op for that field. To clear a configured value, call resetTraigentConfig() and then configure the values you still want.

    Configuration precedence is:

    1. environment variables
    2. configure() / initialize() defaults
    3. per-client or per-call options

    Supported environment variables are:

    • TRAIGENT_BACKEND_URL
    • TRAIGENT_API_URL
    • TRAIGENT_API_KEY
    • TRAIGENT_JWT_TOKEN
    • TRAIGENT_TENANT_ID
    • TRAIGENT_PROJECT_ID
    • TRAIGENT_OFFLINE_MODE
    • TRAIGENT_RELEASE
    • TRAIGENT_ENVIRONMENT
    • TRAIGENT_OBSERVABILITY_PRIVACY_MODE
    • TRAIGENT_TVAR_OBSERVATION
    • TRAIGENT_PARENT_PID
    • TRAIGENT_IDLE_TIMEOUT_MS
    • TRAIGENT_ENABLE_EXPERIMENTAL_RUNTIME_SEAMLESS

    Client config resolves as environment defaults, then configured defaults from configure(...) / initialize(...), then per-client or per-call options. Hybrid optimize/session helpers also accept explicit backendUrl and apiKey options over env defaults.

    Root Authoring Aliases¶

    Python-style authoring aliases are available from @traigent/sdk:

    • Range(min, max, options?)
    • IntRange(min, max, options?)
    • LogRange(min, max, options?)
    • Choices(values, options?)
    • ConfigSpace(configurationSpace, options?)
    • when(predicate).then(predicate)
    • implies(condition, consequence)
    • requires(predicate)
    • require(predicate)

    These are lightweight wrappers over the existing JS param.* and predicate constraint model. They are not a full port of Python's TVL expression-builder classes.

    requires(predicate) and require(predicate) are authoring markers that return the predicate unchanged. They exist for readability and grep-ability, not for runtime validation or transformation.

    Prefer requires(predicate) in JS code. require(predicate) is kept as a Python migration alias, but it can shadow CommonJS/global require in mixed CJS files and may trip downstream no-shadow lint rules.

    For TypeScript annotations, use ConfigSpaceDefinition as the exported type name. ConfigSpace(...) is the value-side Python authoring helper.

    Schema And Casing¶

    Backend DTO types are generated from ../TraigentSchema/traigent_schema/schemas by:

    npm run generate:schema-types
npm run check:schema-types
npm run generate:version
npm run check:version

    npm run build, npm run build:sdk, and prepublishOnly regenerate the SDK version constant before emitting bundles so local builds do not ship stale User-Agent telemetry.

    The generated schema types are available from @traigent/sdk/schema. Public JS client methods expose camelCase types and convert to snake_case at the HTTP boundary.

    Unknown request fields pass through after casing conversion except explicit security-sensitive keys such as nested auth tokens and tenant overrides. null values are preserved.

    Most backend client methods expect the Python-style { data: ... } response envelope and unwrap it. Typed measure creation/update intentionally returns the raw backend object because the Python measures path also bypasses the product client envelope convention.

    Observability Parity¶

    ObservabilityClient emits the same trace/observation field shape as the Python client: input_data, output_data, prompt_reference, correlation_ids, token counts, latency/cost fields, nested observations, and trace/session metadata. JS tests compare the emitted payload shape before it is submitted to the ingest endpoint.

    Python develop also exposes default-client helpers (get_default_observability_client, set_default_observability_client) and an observe wrapper. Those are intentionally deferred out of JS 0.2.0 and tracked for 0.3.0 in Traigent/traigent-js#38 because they add new runtime semantics (process-global state and decorator-style instrumentation) beyond the client parity release. See Deferred Client Policy for the gating criteria.

    Deferred Client Policy¶

    This section is the single source of truth for deferred or REST-only policy decisions covered by the schema-owned manifest and the linked JS tracking issues. JS 0.2.x intentionally does not implement the listed surfaces, and the parity manifest in TraigentSchema records each one as deferred-schema or deferred-backlog.

    The general policy is the same for all three: the surface stays a stub (or absent) in @traigent/sdk until the upstream contract is stable. Stability requires, at minimum:

    1. an endpoint/contract owned by TraigentSchema/ (request/response shapes under TraigentSchema/traigent_schema/schemas),
    2. matching Python implementation in Traigent/traigent/ exported from traigent.__all__,
    3. parity-manifest reclassification from deferred-* to matched in ../TraigentSchema/parity/python-js-sdk.json,
    4. a JS minor release window large enough to ship the implementation with tests and a cross-SDK oracle update under CROSS_SDK_VALIDATION.md.

    Reclassifying any of these to matched without all four conditions is a release-blocking error.

    EnterpriseAdminClient (Traigent/traigent-js#30)¶

    • Current JS state: exported runtime symbol; every method throws UnsupportedActionError with a "use the Traigent REST API" hint.
    • Policy: REST-only until the enterprise admin surface (tenants, tenant memberships, tenant SSO config) is owned by a TraigentSchema contract that both Python and JS can consume. The JS stub remains throwing; the export is not removed because Python re-exports the symbol at package root and the parity manifest requires the name to exist.
    • Migration guidance for SDK users: call the Traigent REST API directly for tenant CRUD, tenant memberships, and tenant SSO config. The OpenAPI reference for the platform admin surface is owned by TraigentBackend/ and the schemas live under TraigentSchema/traigent_schema/schemas. The SDK does not bundle a wrapper, fallback HTTP client, or REST helper for these endpoints in 0.2.x.
    • Release gating: implementation is not in scope for any 0.2.x release. A future minor release may pick up the surface only after the four stability conditions above are met. Until then, do not add new methods, rename methods, or change the stub error message in a way that could read as a real implementation.

    BenchmarkClient (Traigent/traigent-js#31)¶

    • Current JS state: exported runtime symbol with two stub methods, generate(...) and generateSync(...), both throwing UnsupportedActionError referencing the deferred dataset-generation endpoint.
    • Policy: schema-deferred until both (a) the Python and JS public surface agree on whether benchmark functionality ships as BenchmarkClient, a differently named client, or no root client at all, and (b) TraigentSchema owns the dataset-generation request/response contract. The JS export is retained so the parity-manifest guard does not flap when Python toggles the symbol on traigent.__all__.
    • Migration guidance for SDK users: until the contract lands, use the existing EvaluationClient and the Dataset / EvaluationExample runtime classes to assemble evaluation datasets locally, or call the platform REST surface for any pre-built datasets the backend exposes. Do not depend on any specific BenchmarkClient method name, return type, or import path in application code; they may change when the naming decision lands.
    • Release gating: no implementation in 0.2.x. The first JS release that ships a working benchmark client must (a) land after the TraigentSchema-owned endpoint, (b) match the agreed product name from Traigent/traigent-js#31, and (c) update the parity manifest entry from deferred-schema to matched in the same family of PRs.

    Default observability helpers and span wrappers (Traigent/traigent-js#38)¶

    • Current JS state: not implemented and not exported. The exported ObservabilityClient class is the only entry point; consumers construct it explicitly per use site.
    • Policy: deferred to 0.3.0 at the earliest. The Python helpers get_default_observability_client, set_default_observability_client, and observe, plus the broader add_agent_span style helper family, introduce process-global state and decorator/span instrumentation semantics that the 0.2.0 client-parity release intentionally avoided. They will not be added in 0.2.x even as stubs, because adding throwing global accessors would commit the SDK to a runtime shape before the semantics are agreed.
    • Migration guidance for SDK users: construct an ObservabilityClient explicitly at module load time and pass it through application code, or call the observability REST routes through the existing client. For decorator-style instrumentation, wrap the target function manually with startTrace / recordObservation / endTrace; do not rely on the Python-side global helper shape carrying over verbatim.
    • Release gating: target window is @traigent/sdk 0.3.0, contingent on agreement about (a) where the default client lives (per-process singleton vs. per-context via the existing TrialContext), (b) whether observe binds to that default or to a passed-in client, and (c) error and async semantics for the wrapped function. None of those decisions are settled in 0.2.0, so 0.3.0 cannot land before they are.

    Assisted authoring and onboarding backlog¶

    • Current JS state: Python-only today. JS does not expose catalog recommendation APIs, strategy presets, advisory selection helpers, or an onboarding CLI.
    • Policy: deferred-backlog. These surfaces combine product guidance, benchmark/objective recommendation, and CLI workflow semantics; they should only move into JS after the product contract, schema classification, and cross-SDK oracle coverage are settled.
    • Migration guidance for SDK users: author JS configuration spaces directly with param.*, ConfigSpace(...), Range(...), IntRange(...), LogRange(...), and Choices(...), or use platform documentation/workflows outside this package for recommendation and onboarding guidance.

    Snapshot stability¶

    The runtime API snapshot at tests/integration/fixtures/api-surface.snapshot.json and the parity-clients test at tests/unit/platform/parity-clients.test.ts are the load-bearing checks for the policy in this section. If you reclassify any of the three surfaces, update the snapshot and the test in the same PR.

    Metric Edge Cases¶

    The optimization surface follows these parity rules:

    • evaluation.metric is an accuracy-style primary metric named accuracy.
    • evaluation.metrics / evaluation.metricFunctions derive the first metric name as the primary metric when objectives are omitted.
    • Boolean metric returns are normalized to 1 or 0.
    • null metric values are preserved through aggregation; objectives still require a finite numeric value for best-config selection.
    • A conflicting metric definition means the same metric key is provided by both evaluation.metrics and evaluation.metricFunctions.

    Error Mapping¶

    Python exception JS error Retryable
    AuthenticationError AuthenticationError no
    ClientError ClientError only 408, 429, or 5xx
    TraigentConnectionError TraigentConnectionError yes
    validation/config errors ValidationError no
    unsupported operation UnsupportedActionError no

    For rejected HTTP authentication/authorization responses, the parsed or raw server response body is available at error.details.body.

    Validation Command¶

    Use the bundled CI command for parity changes:

    npm run test:ci

    It includes schema freshness, SDK-version freshness, the schema-owned Python/JS parity manifest check, format, lint, typecheck, coverage, build, and package smoke checks for every subpath export in both ESM and CJS.

    The authoritative manifest lives in the sibling TraigentSchema repo at:

    ../TraigentSchema/parity/python-js-sdk.json

    The JS checker pins only the Python SHA and verifies this package satisfies the manifest-required root exports:

    npm run build:sdk
npm run check:parity-manifest

    Manifest refresh cadence:

    • refresh on every Python public SDK release candidate or release branch
    • refresh in the same PR family as any Python change to traigent.__all__
    • refresh before every JS minor parity release
    • if neither SDK releases for a quarter, run a quarterly drift check and open a no-op or update PR documenting the result

    TVL fixtures remain owned by the sibling ../tvl repo. Do not copy divergent fixtures into this package; vendor them only via a script with checksum verification when needed.
